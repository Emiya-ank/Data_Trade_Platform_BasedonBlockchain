# 模块化谓词电路设计文档

## 📋 概述

模块化谓词电路允许构建多个独立的证明电路，每个电路验证不同的数据属性，但都以相同的**数据承诺 `CommitX`** 作为公开锚点。

这实现了 ZKCPlus 框架的核心设计：**一份数据，多个谓词，统一聚合**。

## 🔧 谓词接口设计

### 通用接口：`PredicateCircuit`

```go
type PredicateCircuit interface {
    // 实现电路约束逻辑
    Define(api frontend.API) error
    
    // 返回该谓词的公开输入信息
    GetPublicInputs() map[string]interface{}
    
    // 返回谓词名称（便于日志和聚合追踪）
    GetPredicateName() string
}
```

所有谓词都**必须**包含以下公开字段：
```go
CommitX frontend.Variable `gnark:",public"` // 数据承诺（跨证明链接）
```

## 📐 三个内置谓词详解

### 1️⃣ 范围谓词 (RangePredicate)

**用途**：验证 $a \leq x[i] \leq b$

**应用场景**
- 年龄/金额范围验证
- 信用评分范围检查
- 数值约束验证

**电路约束**

| 约束 | 描述 | 数学形式 |
|------|------|---------|
| 索引有效性 | 验证 $\text{idx} < \text{textLen}$ | $0 \leq \text{idx} < |x|$ |
| 下界检查 | 验证 $x[\text{idx}] \geq a$ | $x[\text{idx}] - a \geq 0$ |
| 上界检查 | 验证 $x[\text{idx}] \leq b$ | $b - x[\text{idx}] \geq 0$ |

**约束数量**：~150-200（包括二进制分解）

**代码示例**

```go
// 电路定义
type RangePredicateCircuit struct {
    Plaintext [MAX_N]frontend.Variable      // 私有
    TextLen   frontend.Variable             // 私有
    CommitX   frontend.Variable `gnark:",public"`  // 公开
    RangeMin  frontend.Variable `gnark:",public"`  // a
    RangeMax  frontend.Variable `gnark:",public"`  // b
    TargetIdx frontend.Variable `gnark:",public"`  // i
}

// 约束实现
func (c *RangePredicateCircuit) Define(api frontend.API) error {
    // 1. idx 有效性
    api.AssertIsLessOrEqual(c.TargetIdx, c.TextLen)
    
    // 2. x[idx] >= a
    low_diff := api.Sub(c.Plaintext[0], c.RangeMin)
    bin_low := api.ToBinary(low_diff, 64)
    api.AssertIsZero(bin_low[63])  // 符号位应为 0
    
    // 3. x[idx] <= b
    high_diff := api.Sub(c.RangeMax, c.Plaintext[0])
    bin_high := api.ToBinary(high_diff, 64)
    api.AssertIsZero(bin_high[63])
    
    return nil
}
```

### 2️⃣ 选择性披露谓词 (SelectiveDisclosure)

**用途**：证明 $x' = x \circ b$（隐私保护的子集提取）

**应用场景**
- GDPR 兼容的隐私披露
- 选择性信息共享（论文 §4.3.2）
- 属性选择性验证

**电路约束**

| 约束 | 描述 |
|------|------|
| 掩码布尔性 | $b_i \in \{0, 1\}$ |
| 掩码连续性 | 前 $k$ 个为 1，后面为 0 |
| 披露一致性 | $x'_i = x_i \cdot b_i$ |

**约束数量**：~300-400（包括掩码验证和一致性检查）

**代码示例**

```go
type SelectiveDisclosureCircuit struct {
    Plaintext      [MAX_N]frontend.Variable      // 私有
    Mask           [MAX_N]frontend.Variable      // 私有（b）
    CommitX        frontend.Variable `gnark:",public"`
    DisclosedLen   frontend.Variable `gnark:",public"`  // k
    DisclosedData  [MAX_N]frontend.Variable `gnark:",public"`  // x'
}

func (c *SelectiveDisclosureCircuit) Define(api frontend.API) error {
    // 1. 掩码布尔性
    for i := 0; i < MAX_N; i++ {
        api.AssertIsBoolean(c.Mask[i])
    }
    
    // 2. 掩码连续性 + 3. 披露一致性
    for i := 0; i < MAX_N; i++ {
        isInRange := api.IsZero(api.Sub(c.DisclosedLen, i+1))
        
        // 超出范围的 mask 应为 0
        maskOutRange := api.Mul(c.Mask[i], api.Sub(1, isInRange))
        api.AssertIsZero(maskOutRange)
        
        // 披露数据一致性
        expectedDisclosed := api.Mul(c.Plaintext[i], c.Mask[i])
        api.AssertIsEqual(c.DisclosedData[i], expectedDisclosed)
    }
    
    return nil
}
```

### 3️⃣ 哈希一致性谓词 (HashConsistency)

**用途**：验证 $\text{Poseidon2}(x) = h$

**应用场景**
- 数据完整性验证
- 内容寻址存储验证
- 承诺合约验证

**电路约束**

| 约束 | 描述 |
|------|------|
| 哈希计算 | 通过 Poseidon2 电路计算 |
| 哈希一致性 | $\text{computed\_hash} = h$ |

**约束数量**：~200-300（Poseidon2 轮函数展开）

**代码示例**

```go
type HashConsistencyCircuit struct {
    Plaintext  [MAX_N]frontend.Variable      // 私有
    TextLen    frontend.Variable             // 私有
    CommitX    frontend.Variable `gnark:",public"`
    TargetHash frontend.Variable `gnark:",public"`  // h
}

func (c *HashConsistencyCircuit) Define(api frontend.API) error {
    // 创建 Poseidon2 哈希器
    poseidonHasher, err := poseidon2.NewHasher(api)
    if err != nil {
        return err
    }
    
    // 哈希有效数据
    validElements := make([]frontend.Variable, 0)
    for i := 0; i < MAX_N; i++ {
        validElements = append(validElements, c.Plaintext[i])
    }
    
    // 验证哈希值
    computedHash := poseidonHasher.Hash(validElements...)
    api.AssertIsEqual(computedHash, c.TargetHash)
    
    return nil
}
```

## 🔗 跨谓词链接机制

### CommitX 的关键作用

```
证明方侧：
    原始数据 x
         │
         ├─→ [Pedersen] → CommitX
         │
         ├─→ φ₁(x) (验证过程中使用私有 x)
         ├─→ φ₂(x) (不同谓词，同一数据)
         └─→ φ₃(x)

验证方侧：
    仅收到 CommitX、π₁、π₂、π₃、公开参数
    
    验证步骤：
    1. 验证 π₁ w.r.t. pk₁、CommitX
    2. 验证 π₂ w.r.t. pk₂、CommitX
    3. 验证 π₃ w.r.t. pk₃、CommitX
    
    如果所有验证成功 → 确认数据 x 将所有谓词
```

**关键特性**
- CommitX 是所有谓词的共同公开参考点
- 即使验证方不知道 x，也能验证数据属性
- 单个 CommitX 可匹配无限多个谓词

## 💡 设计要点总结

1. **统一接口**：所有谓词实现 `PredicateCircuit`
2. **共同锚点**：所有谓词以 `CommitX` 为公开链接
3. **独立 Setup**：各谓词独立 Setup，生成各自 pk/vk，共享通用 SRS
4. **模块化组合**：可任意组合谓词，便于扩展
5. **聚合友好**：设计便于后续 SnarkPack 聚合
